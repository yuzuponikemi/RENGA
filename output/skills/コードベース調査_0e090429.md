# 🔷 スキル定義: コードベース構造調査

**説明**: リポジトリの構造・実装を調査し報告するスキル
**ステータス**: 🔒 審査中 | **ソース数**: 6 件 | **カテゴリ**: コードベース調査

---

## テンプレートプロンプト（抽象）

```
{target_directory} を調査し、以下を簡潔に報告してください（合計{word_limit}語以内）：

1. **{topic1}**：{detail1}
2. **{topic2}**：{detail2}
3. **{topic3}**：{detail3}
4. **{topic4}**：{detail4}
5. **{topic5}**：{detail5}

ファイル全体を読む必要はなく、grep/findでファイル名とエントリ関数を特定して報告してください。判断や提案は不要、事実だけ。
```

## 変数一覧

- `{{target_directory}}`: 調査対象のディレクトリパス（例: あなたのプロジェクトのルートディレクトリ）
- `{{word_limit}}`: 回答の最大語数（例: 300, 400, 500）
- `{{topic1}}`: 調査項目1の名前（例: エントリポイント）
- `{{detail1}}`: 調査項目1の詳細説明（例: メインのmain.pyやrun_pipeline.pyの場所と内容）
- `{{topic2}}`: 調査項目2の名前（例: グラフ/ワークフロー定義）
- `{{detail2}}`: 調査項目2の詳細説明（例: LangGraphのStateGraphファイルとノード遷移）
- `{{topic3}}`: 調査項目3の名前（例: 主要クラスの中身）
- `{{detail3}}`: 調査項目3の詳細説明（例: BaseAgentクラスのinvoke()メソッドのシグネチャとロジック）
- `{{topic4}}`: 調査項目4の名前（例: ツール実装）
- `{{detail4}}`: 調査項目4の詳細説明（例: 使用している外部ツールとその実装ファイル）
- `{{topic5}}`: 調査項目5の名前（例: テストカバレッジ）
- `{{detail5}}`: 調査項目5の詳細説明（例: tests/ディレクトリの有無と内容の量）

## 活用シーン

- 新しいリポジトリを初めて調査する際のベースプロンプトとして
- 複数リポジトリの横断比較調査を効率化するテンプレートとして
- 特定の機能（エージェントループ、ツール使用等）の有無を確認する調査に

**実際のプロンプト**

### インスタンス 2: Multiagents詳細調査

> 元プロンプト: *{target_directory} を深く調査し、以下を簡潔に報告してください（合計{word_limit}語以内）。Phase 6 では「内側のエージェントロジックを {別プロジェクト名} の {エージェント名} に置換」したい。*

**変数の値**
  - `target_directory`: 調査対象のディレクトリパス（例: /Users/yourname/source/personal/Multiagents/）
  - `word_limit`: 500
  - `topic1`: BaseAgentクラスの中身
  - `detail1`: src/multiagents/agents/base.pyのinvoke()メソッドのシグネチャとロジック概要（LLM呼ぶだけ？ループ？tool呼び出し？）
  - `topic2`: persona実装の例
  - `detail2`: Coder, Ghost, Analystの各agent実装。LLM単発呼び出しor多段ループ？
  - `topic3`: graph構造
  - `detail3`: graphs/lab_meeting.pyのStateGraph（ノード遷移を箇条書きで）
  - `topic4`: tool実装
  - `detail4`: JiraClient以外のツール有無。各personaがtoolを使う設計かテキストのみか
  - `topic5`: persona injectionの仕組み
  - `detail5`: dynamic persona injectionの実態。config/dataで動的切り替え可能かハードコードか

**実際のプロンプト**
```
{target_directory} を深く調査し、以下を簡潔に報告してください（合計{word_limit}語以内）。Phase 6 では「内側のエージェントロジックを {別プロジェクト名} の {エージェント名} に置換」したい。それが現実的かどうか判定するための情報を集めたい。

1. **BaseAgent クラスの中身**：`src/multiagents/agents/base.py` 等にあるはず。`invoke()` メソッドのシグネチャとロジック概要（LLM呼ぶだけ？ ループしてる？ tool 呼び出し？）。
2. **persona実装の例**：Coder, Ghost, Analyst の各 agent 実装。実体はLLM単発呼び出し or 多段ループ？
3. **graph 構造**：`graphs/lab_meeting.py` の StateGraph（ノード遷移を箇条書きで簡潔に）。
4. **tool 実装**：JiraClient 以外にツールがあるか。各 persona が tool を使う設計か、それともテキストだけで会話するか。
5. **persona injection の仕組み**：「dynamic persona injection」と説明されているので、persona は config/data で動的に切り替え可能か、ハードコードか。
6. **テスト**：tests/ の量と内容（量だけ）。
7. **公開API**：他リポから `from multiagents import ...` で使う想定か、CLI 専用か。

判断や提案は不要。「{エージェント名}に置き換え可能か」も判断不要。事実だけ。grep/find ベースで速く。
```

### インスタンス 3: local-deepR調査

> 元プロンプト: *{target_directory} を調査し、以下を簡潔に報告してください（{word_limit}語以内）：  1. **search provider 実装のファイルパス**：Perplexity と SearXNG（あれば他も）。*

**変数の値**
  - `target_directory`: 調査対象のディレクトリパス（例: /Users/yourname/source/personal/local-deepR/）
  - `word_limit`: 400
  - `topic1`: search provider実装のファイルパス
  - `detail1`: PerplexityとSearXNGの関数名/クラス名
  - `topic2`: provider実装の依存
  - `detail2`: 環境変数、API key、HTTPクライアント、langchain連携の有無
  - `topic3`: 別プロジェクトに欠けている価値ある部分
  - `detail3`: 特殊なreflection logic、独自citation抽出、独自プロンプト等
  - `topic4`: エントリポイント
  - `detail4`: CLI、StateGraph、関数API。retirement時の置き換えマッピング
  - `topic5`: テストカバレッジ
  - `detail5`: tests/ディレクトリの有無と内容の量

**実際のプロンプト**
```
{target_directory} を調査し、以下を簡潔に報告してください（{word_limit}語以内）：

1. **search provider 実装のファイルパス**：Perplexity と SearXNG（あれば他も）。それぞれの関数名/クラス名。
2. **provider 実装の依存**：環境変数、API key、HTTPクライアント、langchain連携の有無。
3. **provider 以外で {別プロジェクト名} に欠けていそうな価値ある部分**：例えば特殊な reflection logic、独自の citation 抽出、独自プロンプト等があれば指摘。

5. **テストカバレッジ**：tests/ ディレクトリの有無と内容（量だけ）。

判断や提案は不要。事実だけ。ファイル全部読まなくてよいので、grep / find で速く。
```
