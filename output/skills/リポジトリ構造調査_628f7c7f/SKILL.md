# リポジトリ構造調査

**説明**: 指定リポジトリの構造・モジュール・データフローを体系的に調査する
**ステータス**: ✅ 公開中 | **ソース数**: 8 件 | **ユニークユーザー**: 3 人 | **カテゴリ**: リポジトリ構造調査

---

## テンプレートプロンプト

```
{リポジトリパス} のリポジトリ構造を詳細に調査してください。

以下を全て調べてください：

1. **ディレクトリ構造** — 全ファイル一覧（{ファイル拡張子}）
2. **主要モジュールの役割** — {主要ディレクトリ} 以下の各サブパッケージの目的
3. **パイプラインの全体像** — {エントリポイント} → {中間モジュール} → {出力先} の流れ
4. **データフロー** — 入力（{入力形式}）→ 処理 → 出力（{出力形式}）の流れ
5. **外部システムとの関係** — {外部システム名} との接続点と担当モジュール
6. **他リポジトリとの接点** — {関連リポジトリ} を import/参照しているか
7. **設定・環境変数** — {設定ファイルパス}, 環境変数一覧
8. **現在の問題点** — {調査観点}

報告形式：Markdown で各セクションを明確に分け、コードブロックで重要なコードを引用。

目的：{調査目的}
```

## 変数一覧

- `{{リポジトリパス}}`: 調査対象のリポジトリの絶対パスまたは相対パス
- `{{ファイル拡張子}}`: 調査対象のファイル拡張子（例: .py, .md, .toml）
- `{{主要ディレクトリ}}`: 主要なソースコードディレクトリ名（例: src/, app/）
- `{{エントリポイント}}`: パイプラインの開始点となるファイルまたはディレクトリ
- `{{中間モジュール}}`: パイプラインの中間処理モジュール
- `{{出力先}}`: 最終的な出力先（例: データベース, ファイル出力）
- `{{入力形式}}`: システムが受け付ける入力形式（例: URL, ファイル形式）
- `{{出力形式}}`: システムが生成する出力形式（例: 記事, データベースレコード）
- `{{外部システム名}}`: 連携する外部システム（例: データベース, ベクトルストア）
- `{{関連リポジトリ}}`: 関連する他のリポジトリ名
- `{{設定ファイルパス}}`: 設定ファイルのパス（例: pyproject.toml, config/）
- `{{調査観点}}`: 特に注目すべき問題点や調査観点
- `{{調査目的}}`: この調査の目的

## 活用シーン

- 新規参画メンバーが既存リポジトリの全体像を把握するためのオンボーディング調査
- 複数リポジトリ間の統合や置き換えを検討する前の現状分析
- 特定機能（RAG、エージェントループ、外部連携）の実装詳細を把握するための調査

### インスタンス 2: test-smith RAG調査

> 元プロンプト: *指定リポジトリのRAGとドキュメント処理に関する実装を包括的に調査してください。  以下の観点で調査： 1. ドキュメント取り込みパイプライン（取り込みスクリプトディレクトリ配下） 2. 前処理システム（前処理モジュール配下） 3. RAG*

**変数の値**
  - `リポジトリパス`: 指定されたリポジトリのパス
  - `ファイル拡張子`: .py, .md, .toml, .yaml
  - `主要ディレクトリ`: ソースコードディレクトリ
  - `エントリポイント`: 取り込みスクリプトディレクトリ
  - `中間モジュール`: 前処理モジュール → ノードモジュール
  - `出力先`: ベクトルデータベース検索結果
  - `入力形式`: ドキュメントファイル
  - `出力形式`: ベクトルデータベース検索結果
  - `外部システム名`: ベクトルデータベース
  - `関連リポジトリ`: 関連リポジトリA, 関連リポジトリB
  - `設定ファイルパス`: プロジェクト設定ファイル, 設定ディレクトリ
  - `調査観点`: チャンキング戦略の品質、評価メトリクスの有無
  - `調査目的`: RAGパイプラインを理解し、品質改善のための現状把握を行うため

### インスタンス 3: エージェント層調査

> 元プロンプト: *指定された親ディレクトリ配下の以下のリポジトリを調査し、各リポの「エージェント的ロジック」（多段判断ループ、ReAct、Planner-Executor、Tool-use、persona switching 等）の有無と中身を簡潔に報告して*

**変数の値**
  - `リポジトリパス`: 指定された親ディレクトリ
  - `ファイル拡張子`: .py, .md, .toml
  - `主要ディレクトリ`: 各リポジトリのソースコードディレクトリまたはエージェントディレクトリ
  - `エントリポイント`: 各リポジトリのメインファイル
  - `中間モジュール`: 各リポジトリのグラフ定義ファイル
  - `出力先`: 各リポジトリの出力形式
  - `入力形式`: 各リポジトリの入力形式
  - `出力形式`: 各リポジトリの出力形式
  - `外部システム名`: LLM実行環境, グラフフレームワーク, ベクトルデータベース
  - `関連リポジトリ`: 統合対象リポジトリ
  - `設定ファイルパス`: プロジェクト設定ファイル
  - `調査観点`: エージェントループの有無、LLM経路、ツール定義形式、公開APIの有無
  - `調査目的`: 統合対象リポジトリと統合可能なエージェントロジックの有無を把握するため

**実際のプロンプト**
```
{リポジトリパス} 配下の以下のリポジトリを調査し、各リポの「エージェント的ロジック」（多段判断ループ、ReAct、Planner-Executor、Tool-use、persona switching 等）の有無と中身を簡潔に報告してください。合計600語以内。

**Agent Layer の他リポ（統合候補）**:
1. `{リポジトリ名A}` — 説明A
2. `{リポジトリ名B}` — 説明B

**Knowledge Layer（agent loop が紛れていないか確認）**:
3. `{リポジトリ名C}` — 説明C
4. `{リポジトリ名D}` — 説明D
5. `{リポジトリ名E}` — 説明E

**その他**:
6. `{リポジトリ名F}` — 説明F
7. `{リポジトリ名G}` — 説明G
8. `{リポジトリ名H}` — 説明H

各リポについて：
- **A. 多段ループの所在**：エージェントループが実装されている**ファイルパス＋クラス/関数名**（ある場合のみ）
- **B. 使っているLLM経路**：自前 LLM 抽象か、LangChain/LangGraph か、Anthropic SDK か等
- **C. ツール定義の形式**：Tool subclass パターン / function calling / 自由形式 / なし
- **D. 公開API（importable）か CLI 専用か**

判断や統合提案は不要。事実だけ簡潔に。ファイル全部読まなくて良いので、grep / find で速く。リポが存在しない or 空ならその旨だけ報告。

合計600語上限を厳守。長くしないこと。

目的：{調査目的}
```
