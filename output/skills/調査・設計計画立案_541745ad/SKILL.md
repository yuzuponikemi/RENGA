## 変数一覧

- `{{調査対象}}`: 調査・設計の対象（例: データ構造、API、コミュニティツール）
- `{{調査の観点}}`: 調査の切り口（例: データ構造、API、コミュニティツール）
- `{{調査対象の詳細}}`: 調査対象の具体的な説明（例: データ形式、ディレクトリ構造、含まれる情報）
- `{{観点1}}`: 最初の調査観点（例: データ形式と構造）
- `{{観点2}}`: 2つ目の調査観点（例: APIの有無と仕様）
- `{{観点3}}`: 3つ目の調査観点（例: コミュニティツールや代替手段）
- `{{調査方法1}}`: 1つ目の調査方法（例: 公式ドキュメントの確認）
- `{{調査方法2}}`: 2つ目の調査方法（例: GitHub検索）
- `{{調査方法3}}`: 3つ目の調査方法（例: ネット検索やフォーラム調査）
- `{{出力形式}}`: 成果物の形式（例: 詳細な実装計画、コードスニペット付きレポート）
- `{{出力の制約}}`: 出力の制約条件（例: 300語以内、ファイルは作成しない）
- `{{既存の前提条件や制約事項}}`: 既存のコードベースや環境の前提条件（例: Python 3.11+、依存関係ゼロ）
- `{{設計計画に必要な要素}}`: 設計計画に含めるべき要素（例: 新規ファイル、CLIサブコマンド、設定変更）
- `{{成果物の種類}}`: 最終的な成果物の種類（例: 実装計画書、設計ドキュメント）

### インスタンス 1: Gemini会話スレッド調査

> 元プロンプト: *Thoroughly research how to track conversation continuity/threading in chat history. The user has exported data via the s*

**変数の値**
  - `調査対象`: 会話履歴のスレッド管理方法
  - `調査の観点`: データ構造・API・コミュニティツール
  - `調査対象の詳細`: 会話履歴におけるスレッド/連続性の追跡方法。ユーザーはデータエクスポート機能からJSON形式でデータをエクスポートしたが、各プロンプトが独立したエントリとして保存されており、会話スレッドIDが存在しない。
  - `観点1`: データエクスポートの代替手段：会話スレッドを保持する別のエクスポート形式やカテゴリの有無
  - `観点2`: API：スレッドID付きの会話履歴を提供するAPIの有無
  - `観点3`: Webインターフェースのスクレイピング：Webサイトからの会話履歴取得方法、隠しAPIエンドポイント、ブラウザDevToolsでの調査
  - `観点4`: コミュニティツール：GitHub上の会話エクスポートツール、スレッド再構築プロジェクト、ブラウザ拡張機能
  - `観点5`: エンタープライズ向けAPI：組織向け会話ログAPI
  - `観点6`: リバースエンジニアリング：Webアプリの内部API構造、会話一覧エンドポイント、URL内のスレッドID
  - `調査方法1`: GitHub、Reddit、Stack Overflow、ブログ記事、ドキュメントの徹底検索
  - `調査方法2`: 最新情報に焦点
  - `調査方法3`: 各角度について具体的なツール名やエンドポイントを特定
  - `出力形式`: 各調査観点ごとに具体的な発見事項を箇条書きで報告、実現可能性の高い方法を優先順位付け、コードスニペットやAPIエンドポイントの具体例を含む
  - `出力の制約`: なし
  - `既存の前提条件や制約事項`: ユーザーはすでにデータエクスポート機能でデータをJSON形式でエクスポート済み。多数のエントリあり、各エントリはタイトル、タイムスタンプ、HTMLレスポンスを含む。会話グループ化情報は一切なし。
  - `設計計画に必要な要素`: スレッド再構築の実装方法、エクスポートデータとスクレイピングデータのマッチング方式（タイムスタンプマッチング、テキストマッチング）、必要な新規ファイルとその責務、エラーハンドリング戦略
  - `成果物の種類`: 会話スレッド再構築の実装計画書

### インスタンス 2: Gemini Webクローラ設計

> 元プロンプト: *Design an implementation plan for adding a "Web Crawler" module to the tool. Here is the full context:*

**変数の値**
  - `調査対象`: ツールへのWebクローラモジュール追加
  - `調査の観点`: アーキテクチャ・実装手順・依存関係
  - `調査対象の詳細`: ツールに「Webクローラ」モジュールを追加するための実装計画。現在のコードベースはPython 3.11+、ゼロ外部依存（本番環境）、特定のビルドシステム、パッケージ管理、CLIサブコマンド、既存モジュール群、テストフレームワーク、特定の互換Markdown出力。
  - `観点1`: アーキテクチャ設計：新規ファイルの責務と配置、既存ファイルへの変更点
  - `観点2`: 実装手順：各ステップの詳細、ブラウザ自動化ツールの導入方法、認証状態の永続化
  - `観点3`: 依存関係管理：設定ファイルの変更、ブラウザ自動化ツールのバージョン選定
  - `調査方法1`: 既存コードベースの構造分析
  - `調査方法2`: ブラウザ自動化ツールの公式ドキュメント確認
  - `調査方法3`: ブラウザ自動化のベストプラクティス調査
  - `出力形式`: ファイル単位の変更リスト、各ステップの具体的なコード構造、CLIサブコマンドの設計
  - `出力の制約`: なし
  - `既存の前提条件や制約事項`: Python 3.11+、現在ゼロ外部依存（本番環境）、特定のビルドシステム、特定のパッケージ管理、CLIエントリ（argparseサブコマンド）、既存モジュール群、テストフレームワーク（tests/ディレクトリ）、出力：特定の互換Markdownファイル、現在のアーキテクチャ：セッションログを読み込み、オブジェクトにパース、Markdownに変換、受信箱に書き込み。鮮度ゲートと重複排除ゲートあり。
  - `設計計画に必要な要素`: 新規ファイルとその責務、新規CLIサブコマンド、設定追加（クッキー保存パス、クローラ固有設定）、エクスポートデータとクロールデータのマッチング方式（タイムスタンプマッチング、テキストマッチング）、ブラウザ自動化ツール状態永続化アプローチ、エラーハンドリング（期限切れセッション、DOM変更など）、テスト戦略、既存ファイルの変更点
  - `成果物の種類`: 詳細なステップバイステップの実装計画書（ファイル単位の変更リスト付き）

### インスタンス 3: MLX Gemma4推論調査

> 元プロンプト: *Research how to use MLX with a specific model (Multi-Token Prediction / Speculative Decoding) on Apple Silicon Mac. Spec*

  - `調査対象`: MLXを使用した特定モデルのApple Silicon Macでの実行方法
