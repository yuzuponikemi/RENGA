# 🔷 スキル定義: 逆算型タスク設計

**説明**: 目標から逆算して段階的・持続的な改善を設計するスキル
**ステータス**: ✅ 公開中 | **ソース数**: 11 件 | **カテゴリ**: 逆算型タスク設計

---

## テンプレートプロンプト（抽象）

```
以下の目標から逆算して、{作業対象}に対して{品質基準}を満たすように{改善内容}を設計してください。特に{重要ポイント}に注意し、{段階的アプローチ}で進められるようにし、{成果物形式}として出力してください。また、{追加要件}も考慮してください。
```

## 変数一覧

- `{{作業対象}}`: 改善・設計の対象（例: ポッドキャスト要約、カリキュラム、レポート生成）
- `{{品質基準}}`: 求める品質レベル（例: 論点10個以上、エンジニアがデバッグ可能、飽きさせない構成）
- `{{改善内容}}`: 具体的な改善事項（例: エンティティ抽出の質向上、逐次経過確認、3段階の品質）
- `{{重要ポイント}}`: 特に注意すべき点（例: 構造の繰り返し回避、重複防止、メトリクス設計）
- `{{段階的アプローチ}}`: 段階的な進め方（例: 動作確認→ロジック確認→本番、恐怖→ワクワクの順）
- `{{成果物形式}}`: 最終的な出力形式（例: 詳細なシラバスドキュメント、ターミナルログ付きコード）
- `{{追加要件}}`: その他の条件（例: uvで構築、claude.mdにルール化、optunaのTPE説明を含む）

## 活用シーン

- ポッドキャスト要約の品質を段階的に向上させる設計
- カリキュラムの重複を防ぎつつ拡張可能なシラバス設計
- レポート生成における品質レベル別のモデル配分設計

# 🔶 具体インスタンス

### インスタンス 1: ポッドキャスト要約改善

> 元プロンプト: *でてきた2つの記事をよみました　いい感じなんだけど、2時間のポッドキャストに対しては論点が少ないですね　論点は10個以上キープさせて　本論の長さが3倍くらいになるようにできますか？*

**変数の値**
  - `作業対象`: ポッドキャスト要約
  - `品質基準`: 論点10個以上、本論の長さ3倍、飽きさせない構成
  - `改善内容`: エンティティ抽出の質向上、論点拡充
  - `重要ポイント`: 特定のエピソードを例に質の高いエンティティ抽出
  - `段階的アプローチ`: 前回までのポッドキャスト要約の質をベースラインとして段階的に改善
  - `成果物形式`: 記事
  - `追加要件`: 同じ構造の繰り返しにならない工夫

**実際のプロンプト**
```
以下の目標から逆算して、ポッドキャスト要約に対して論点10個以上、本論の長さ3倍、飽きさせない構成を満たすようにエンティティ抽出の質向上と論点拡充を設計してください。特に特定のエピソードを例に質の高いエンティティ抽出ができるように注意し、前回までのポッドキャスト要約の質をベースラインとして段階的に改善できるようにし、最終的な記事として出力してください。また、同じ構造の繰り返しにならないような工夫も考慮してください。
```

### インスタンス 2: シラバス整理と拡張

> 元プロンプト: *ありがとう　今までにどんなトピックのどういう学習内容を網羅してきたのか、一目で確認できるような詳細なシラバスドキュメントを用意してもらえますかあとで何かコンテンツを追加したりするときに、重複していないかどうかなどの判断材料になるようにしたい*

**変数の値**
  - `作業対象`: カリキュラムのシラバス
  - `品質基準`: 一目で全トピックと学習内容を確認でき、重複がない状態
  - `改善内容`: 詳細なシラバスドキュメントの作成、古い文書の整理
  - `重要ポイント`: コンテンツ追加時の重複チェックができること
  - `段階的アプローチ`: 現状のカリキュラム文書を整理→プロジェクトルールファイルにルール追加→持続的アップデート
  - `成果物形式`: シラバスドキュメント
  - `追加要件`: 古くなったカリキュラム文書の整理

**実際のプロンプト**
```
以下の目標から逆算して、カリキュラムのシラバスに対して一目で全トピックと学習内容を確認でき、重複がない状態を満たすように詳細なシラバスドキュメントの作成と古い文書の整理を設計してください。特にコンテンツ追加時の重複チェックができるように注意し、まず現状のカリキュラム文書を整理し、その後プロジェクトルールファイルにルールを追加することで持続的にアップデートできるようにし、最終的なシラバスドキュメントとして出力してください。また、古くなったカリキュラム文書の整理も考慮してください。
```

### インスタンス 3: レポート品質3段階設計

> 元プロンプト: *いい方針です　このように、レポートのクオリティを３段階に分けて　動作確認用、ロジック含む確認用、本番用　という感じで、ここにも３パターンのモデル配分が選べるようにしておいてもら得ますか*

**変数の値**
  - `作業対象`: レポート生成
  - `品質基準`: 動作確認用・ロジック含む確認用・本番用の3段階
  - `改善内容`: 3パターンのモデル配分を選べる機能
  - `重要ポイント`: クオリティがいらないテストも存在することを考慮
  - `段階的アプローチ`: 動作確認用→ロジック含む確認用→本番用
  - `成果物形式`: 実装コード
  - `追加要件`: 各品質レベルに応じたモデル配分の選択肢

**実際のプロンプト**
```
以下の目標から逆算して、レポート生成に対して動作確認用・ロジック含む確認用・本番用の3段階の品質を満たすように3パターンのモデル配分を選べる機能を設計してください。特にクオリティがいらないテストも存在することを考慮し、各段階で適切なモデルを割り当てられるように注意し、まず動作確認用から始めて段階的に本番用へ移行できるようにし、最終的な実装コードとして出力してください。また、各品質レベルに応じたモデル配分の選択肢も考慮してください。
```
